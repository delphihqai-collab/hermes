# Deal Status Update

**Type:** Output structure
**Applies to:** Reporting on a specific deal's progress to Boss or in session notes
**Trigger keywords:** deal update, status on deal, where is this deal, deal summary, update on prospect, account status, pipeline update
**Times used:** 0
**Last used:** —

## Skill

One deal = one update block. Never bundle multiple deals into a single narrative.

**DEAL:** [Company name — Contact name — Deal value if known]
**STAGE:** [Current stage: Qualified / Discovery / Proposal / Negotiation / Closed Won / Closed Lost]
**LAST ACTION:** [What happened most recently and when]
**NEXT ACTION:** [Exactly what happens next, who owns it, and by when]
**SIGNAL:** [One honest sentence on momentum — is this moving, stalling, or at risk?]
**NEEDS BOSS:** [Specific ask, or "Nothing right now"]

## Rules

- Never include financial figures (contract value, MRR, deal size) in a #pipeline post. Stage movement goes to #pipeline. Numbers go to #financials only. If a deal update contains both, send two separate messages to the correct channels.
- Write SIGNAL honestly. A stalling deal noted early is manageable. A stalling deal noticed late is a loss.
- NEXT ACTION must always have an owner and a date. Vague next actions are a sign the deal has no momentum.
- If NEEDS BOSS is not empty, this block goes to #hermes-chat immediately, not in the next scheduled update.

## Notes

Use this structure in session notes for every active deal, every day. Over time this creates a reliable deal history that survives context resets.
