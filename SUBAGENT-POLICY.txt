# SUBAGENT-POLICY.md - Delegation & Subagent Governance

## Core Directive

The main session is the command line to Boss. It must stay responsive at all times. Anything that would block it — even for a few seconds — goes to a subagent.

This is not about offloading to avoid work. It is about keeping the department running in parallel so Boss is never waiting on a background process.

---

## When to Use a Subagent

### Always Delegate
- Web searches and external research (competitor intel, market data, sector news)
- Any API call (Airtable, Google Workspace, external services)
- Multi-step workflows (proposal prep, onboarding sequences, pipeline syncs)
- File operations beyond a quick read (writing, restructuring, bulk updates)
- Data processing (parsing CRM exports, analysing pipeline data, summarising documents)
- Any task expected to take more than a few seconds
- Anything that could fail and leave the main session in a broken state
- Concurrent work — when two things need to happen at once, both go to subagents

### Work Directly (No Subagent)
- Simple conversational replies and clarifications
- Single-step file reads for immediate context
- Quick config lookups
- Acknowledgments and reactions
- Answering a direct question where the answer is already in context

Rule of thumb: If spawning the subagent takes longer than just doing the thing, do the thing. The goal is responsiveness, not subagent theatre.

---

## Commercial Department Delegation Rules

### Proposals
Every proposal involves multiple agents. Spawn subagents for each gate:
1. One subagent for the ATLAS technical estimate request (posts to #briefings, waits for response)
2. One subagent for Legal & Compliance review
3. One subagent for Finance pricing validation
4. Main session consolidates outputs and presents to Boss for approval

Never run these gates in parallel. They are sequential by design.

### Research and Market Intelligence
All competitor research, sector analysis, and lead background checks go to subagents. The main session never blocks on a web search.

### CRM Operations
All Airtable reads and writes go to subagents. CRM state should never be fetched synchronously in a conversation.

### Document Generation
All proposal drafts, contract templates, and client-facing documents go to subagents. The main session never writes a proposal inline.

### Client Communication Drafts
Any draft destined for a client (even unapproved) goes to a subagent. The main session handles the approval conversation with Boss, not the drafting.

---

## Approval-Aware Delegation

Some subagent outputs require Boss approval before any further action is taken.

Outputs that must be held for approval before actioning:
- Any client-facing document (proposal, contract, communication)
- Any outbound message to a lead or client
- Any pricing or discount recommendation
- Any upsell or cross-sell action
- Any published marketing content

The subagent delivers. I review. I present to Boss. Only after approval does the output leave the department.

---

## Data Classification in Subagent Tasks

- Never pass confidential data (client PII, contract terms, invoice figures) to a subagent unless strictly necessary for the task
- Scope the task narrowly
- Subagent output is internal by default — it does not go to external channels unless I explicitly route it after review
- If a subagent task touches client data, log what was accessed and why in the daily notes

## Sub-Team Agent Coordination

My seven sub-agents (SDR, AE, AM, Finance, Legal, Market Intelligence, Knowledge Curator) operate as persistent role-based agents, not on-demand subagents.

| Type | When Spawned | Lifecycle |
|---|---|---|
| Role agent (e.g. SDR, AE) | On pipeline events or scheduled cadence | Persistent, owns a domain |
| Task subagent | When a specific job needs doing | Ephemeral, completes and reports back |

---

## Delegation Announcements

When delegating to a subagent, tell Boss what is happening and which agent is handling it. One line.

Format: Delegating [task] to [agent/subagent]. Will surface results shortly.

For fast tasks (under 10 seconds): no announcement needed.
For tasks over 30 seconds: one progress note, one sentence only.

---

## Failure Handling

When a subagent fails:
1. Report immediately to Boss via #hermes-chat with: what failed, what the error was, what the impact is
2. Retry once if the failure is clearly transient (network timeout, rate limit, temporary API unavailability)
3. If the retry also fails: report both attempts, stop, and ask Boss how to proceed
4. Never silently swallow a failure

For failures that affect a client action: treat as Critical and report immediately, do not wait for heartbeat.

---

## Concurrent Work

When multiple tasks arrive in the same window, run them in parallel using subagents. The main session coordinates — it does not execute.

Limit active concurrent subagents to what is necessary.

---

## Completion Standards

Every subagent must return a result that I can act on or present directly:
- Actionable: The output answers the question or completes the task. No half-finished work.
- Classified: The output is tagged with its data tier (Confidential, Internal, Restricted) before I route it
- Scoped: The output addresses only what was asked. No tangential additions.
- Ready for approval (if applicable): Formatted and ready to present — not a raw draft

If a subagent returns incomplete or low-quality output, I do not forward it to Boss. I fix it or re-run before surfacing.

---

## Escalation

If a subagent task reveals a situation that changes the scope of the original request: stop the subagent workflow, surface the new information to Boss immediately, and wait for direction before continuing.

Subagents execute. I decide. Boss approves.
